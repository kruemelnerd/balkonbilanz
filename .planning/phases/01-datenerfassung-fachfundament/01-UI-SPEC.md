## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline metric gaps, helper spacing inside dense cards |
| sm | 8px | Label-to-input spacing, chip gaps, stacked helper text |
| md | 16px | Default card padding, page padding, form section gaps |
| lg | 24px | Separation between form header and body, list header and first item |
| xl | 32px | Gap between major capture sections on larger screens |
| 2xl | 48px | Major break between page intro and form/list area when vertical layout expands |
| 3xl | 64px | Reserved for future page-level sections only; do not use inside cards in this phase |

Exceptions: 44px minimum touch target height for all actionable controls; pill badges may use 8px vertical / 16px horizontal internal padding; list cards may use 16px internal padding for dense read-only rows. (source: existing nav/cards + default)

## Layout and Component Inventory

- Page shell: single-column mobile layout with 16px page padding and 16px vertical gaps.
- Section order on mobile: page intro → meter banner (if present) → meter form → meter list → PV form → PV list.
- Primary focal point on mobile: the meter form card is the first high-attention element below the intro, and its primary save CTA is the strongest visual emphasis on first load.
- At `min-width: 720px`: switch to a 2-column grid; keep each form directly adjacent to its corresponding list.
- Every major unit is a card: white surface, 20px radius, 1px border `rgba(31, 41, 55, 0.12)`, 16px padding.
- Read-only list items may nest lighter cards with 16px radius and 16px padding.
- Headers use one `h1` for page, `h2` for each form/list card; support text stays muted.

## Interaction Contract

### Meter form
- Fields in this exact order: Zeitpunkt → OBIS 1.8.0 → OBIS 2.8.0 → Notiz.
- Use `datetime-local` for Zeitpunkt.
- OBIS inputs use decimal keyboard (`inputmode="decimal"`) and stay left-aligned text fields.
- On validation failure, keep all entered values, show a banner plus issue list, and move focus to the first invalid field.
- If decreasing values trigger the meter-change rule, block save and show a prominent banner with CTA label `Zählerwechsel dokumentieren`.
- Edit mode changes heading to `Zählerstand bearbeiten`; secondary action label is `Abbrechen`.

### PV form
- Fields in this exact order: Tag → Ertrag (kWh) → Quelle → Notiz.
- Use `date` input for Tag and block today/future dates at validation level; do not silently clamp values.
- Default Quelle is `manual`; keep it editable as plain text in this phase.
- Editing an existing day is an update of that day, not a second row.

### Lists
- Sort both lists newest first. (source: `01-CONTEXT.md` D-09)
- Each row exposes exactly two secondary actions: `Bearbeiten` and `Löschen`.
- Actions sit at the bottom of each card and remain reachable with 44px minimum height.
- Meter rows show timestamp first, then 1.8.0 and 2.8.0 totals, then optional note.
- PV rows show day first, then kWh value, source, then optional note.

### Validation and messaging
- Use inline issue lists for field-specific problems and a top-of-card banner for cross-record or submission-level blocking states.
- Error messages must state both problem and remedy in plain German, without technical wording. (source: `01-CONTEXT.md` D-10)
- No soft plausibility warnings for merely unusual-but-possible values in Phase 1. (source: `01-CONTEXT.md` D-05)
- Duplicate timestamp/day conflicts are blocking errors, not warnings.

### Destructive interactions
- Deleting a meter row or PV row requires explicit confirmation before mutation.
- Use a native confirm dialog or a dedicated inline two-step confirmation state; do not delete on the first tap.
- Confirmation copy must include the affected date/time so the user can verify the target entry.

### Accessibility and focus
- Every input needs a visible text label; placeholders are optional and never label replacements.
- Banner regions use `role="alert"` or `aria-live="polite"` depending on interruption severity.
- Quick links from Dashboard to `#meter-timestamp` and `#pv-day` must focus the target input container after navigation. (source: current `CaptureView.vue`)
- Keyboard focus state uses the accent color and remains visible against both dominant and secondary surfaces.
