## Layout and Component Inventory

- Route: add `/settings` as the fourth persistent bottom-nav destination with label `Einstellungen`.
- Page shell: single-column mobile layout with 16px horizontal padding, 16px top padding and 24px gaps between major cards.
- Section order on mobile: page intro/privacy note → Einstellungen & Annahmen card → Tarifhistorie card → Speicherberater card → Backup & Restore card → App-Info card.
- Rationale for order: settings and tariff assumptions must be editable before advisor results are trusted; backup remains visible but lower in the scan because it is episodic, not daily.
- At `min-width: 720px`: use a 2-column grid; left column = settings + tariffs, right column = advisor + backup; keep App-Info full width below.
- Every major unit is a card: white surface, 20px radius, 1px border `rgba(31, 41, 55, 0.12)`, 16px padding.
- Nested structures use lighter sub-cards: 16px radius, 16px padding, 12px internal gaps.
- Use exactly one `h1` (`Einstellungen`) and `h2` per major card.
- Each card header may contain one short helper sentence; keep helper copy to max 2 lines on 390px mobile width.

## Interaction Contract

### Einstellungen & Annahmen
- Fields in this exact order: Strompreis (EUR/kWh) → Einspeisevergütung (EUR/kWh) → Datenqualitätsmodus.
- Defaults are fixed: `0.305 EUR/kWh` for Strompreis and `0 EUR/kWh` for Einspeisevergütung. (source: `SET-01`)
- Data quality mode uses a 3-option segmented control in this exact order: `relaxed`, `balanced`, `strict`.
- `balanced` is the default selected mode unless persisted settings already exist. (source: sensible default)
- Primary action label in this card: `Einstellungen speichern`.
- Saving keeps the user on the page, shows inline success feedback (`Einstellungen lokal gespeichert.`), and immediately refreshes dependent advisor context without requiring a remount.

### Tarifhistorie
- Card title: `Tarifhistorie`.
- Place the add/edit form above the list so the creation path is visible without scrolling the history first.
- Field order: Gültig ab → optional Gültig bis → Strompreis (EUR/kWh) → Speichern.
- The list sorts newest effective period first.
- Every row shows: date range first, then price, then status badge (`Aktiv` or `Vergangen`), then actions.
- Secondary row actions are exactly `Bearbeiten` and `Löschen`.
- Overlapping periods are blocking errors, never warnings.
- Editing switches the form heading to `Tarifperiode bearbeiten`; secondary action label is `Abbrechen`.
- Empty state copy for the list uses the template copy above and must still show the current default price as active fallback.

### Speicherberater
- Card title: `Speicherberater`.
- Show the analysis basis before the inputs: active Analysezeitraum, current data-quality level, current Strompreis and a one-line note that all values are estimates.
- The advisor must use the current analysis period as its basis, not a fixed 30/90-day fallback. (source: `03-DISCUSS-CHECKPOINT.json`)
- The advisor calculation basis must include real analysis outputs from the selected period — at minimum the export/self-consumption relevant analysis result, not only generic period length. (source: `03-VERIFICATION.md` gap)
- Input order: Speicherpreis → Speicherkapazität (kWh) → Wirkungsgrad (%) → Betrachtungszeitraum.
- Default posture is conservative: prefill cautious values, not aggressive sales-like assumptions. (source: `03-DISCUSS-CHECKPOINT.json`)
- Scenario order is fixed: `Konservativ` → `Realistisch` → `Optimistisch` → `Theoretisches Maximum`.
- The four scenarios differ primarily by usable share (`nutzbarer Anteil`). Keep all other user-entered inputs global so comparison remains understandable. (source: `03-DISCUSS-CHECKPOINT.json`)
- Each scenario card must show exactly: scenario name, short description, usable-share label, annual savings in EUR/year, break-even in years.
- `Theoretisches Maximum` represents near-maximum potential, not a physically ideal promise. Phrase it explicitly as an upper bound. (source: `03-DISCUSS-CHECKPOINT.json`)
- Primary action label in this card: `Speicherpotenzial berechnen`.
- When data quality is `poor`, show a prominent warning above the CTA and again above the result grid after calculation.
- Poor-quality warning copy must recommend more capture time before a purchase decision; do not only label the state.
- If no usable analysis basis exists yet, disable the calculate CTA and show an empty-state hint: `Für den Speicherberater brauchst du zuerst Auswertungsdaten aus dem Analysezeitraum.`

### Backup & Restore
- Card title: `Backup & Restore`.
- Export action is always visible first with label `Backup exportieren`.
- Import flow order: Datei auswählen → Datei prüfen → Vorschau anzeigen → Vollständigen Restore bestätigen.
- Preview must show schema version, export timestamp and record counts for settings, tariffs, meter readings and PV entries before any destructive restore action appears.
- Invalid JSON or schema mismatch is a hard stop: show the error in the same card and never expose the destructive restore CTA.
- Use a dedicated inline confirmation panel for restore, not a native confirm dialog.
- Restore CTA label: `Backup jetzt vollständig wiederherstellen`.
- Require an explicit confirmation checkbox with copy `Ich weiß, dass meine aktuellen lokalen Daten vollständig ersetzt werden.` before enabling the destructive restore CTA.
- No merge import in this phase. Copy must say `Vollständiger Restore` consistently. (source: MVP decision in spec)

### App-Info
- Place a low-emphasis info card at the end with local-first/privacy reassurance and current backup reminder.
- Include exactly these two statements in plain German: data stays local in this browser; backups are the only migration path to another device/browser.

### Validation and messaging
- Use inline field errors for single-field issues and a top-of-card banner for cross-field or destructive-flow blockers.
- Error copy must state both problem and next action in plain German.
- Success feedback stays non-modal and card-local; do not use blocking dialogs for successful saves or exports.
- Use `aria-live="polite"` for save/export success and `role="alert"` for restore/validation failures.

### Destructive interactions
- `Tarifperiode löschen` requires explicit confirmation before deletion; native confirm is acceptable here because the target is a single row and reversible via re-entry.
- `Backup jetzt vollständig wiederherstellen` requires preview + checkbox + second-tap confirmation in the same card.
- Destructive elements use the destructive color only after the user enters the confirmation step, not in the neutral browsing state.

### Accessibility and focus
- Every numeric input uses a visible label and `inputmode="decimal"` where appropriate.
- Segmented quality-mode options must expose selected state programmatically (`aria-pressed` or radio semantics).
- File import feedback and preview summary must be keyboard reachable in DOM order after file selection.
- After a blocking validation failure, move focus to the first invalid field or the preview error banner.
- Bottom navigation keeps the active `Einstellungen` destination visibly distinct by both color and text weight, not color alone.

### Additional copy labels
- Settings save success: `Einstellungen lokal gespeichert.`
- Tariff save CTA: `Tarifperiode speichern`
- Tariff edit save CTA: `Tarifänderung speichern`
- Backup preview CTA: `Backup prüfen`
- Invalid backup error: `Diese Datei ist kein gültiges BalkonBilanz-Backup. Deine aktuellen Daten bleiben unverändert.`
- Poor-quality advisor warning: `Die Aussagekraft ist derzeit eingeschränkt. Erfasse länger und vollständiger, bevor du eine Speicherkaufentscheidung triffst.`
