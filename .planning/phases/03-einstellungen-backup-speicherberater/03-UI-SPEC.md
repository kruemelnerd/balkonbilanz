## Screen Composition

Use one mobile-first `Mehr`/`Einstellungen` route with vertically stacked cards in this order:

1. Page header with honest description: settings change future calculations, backup stays local, battery advice is a scenario estimate.
2. `Preis & Qualität` card: Strompreis, Einspeisevergütung, Datenqualitätsmodus.
3. `Tarifperioden` card: existing periods list first, add/edit form second, validation inline.
4. `Backup & Restore` card: export action first, import chooser second, restore preview third.
5. `Speicherberater` card: parameter inputs above, four scenario result cards below.
6. `App-Info` card: local-only/privacy note and schema/app version text.

Desktop enhancement only after mobile is stable: cards may split into two columns from `min-width: 720px`, but action order and card priority stay unchanged.

Primary focal point for this route: the `Speicherberater` action cluster is the explicit visual focal point. Use the full-width accent CTA `Speicher-Szenarien berechnen` as the highest-contrast element in the card, with the four scenario result cards directly following it as the supporting outcome area. Keep all other non-destructive actions visually secondary to this focal point.

## Interaction Contract

- Keep the existing shell pattern: sticky bottom navigation, scrolling content area, rounded cards, explicit save buttons.
- Do **not** use auto-save for settings or battery inputs. Every editable card ends with an explicit primary action.
- Inputs, buttons, segmented options, and file-picker triggers must keep a minimum tap height of **44px**.
- Show validation inline directly under the affected field and repeat critical validation in a card-level alert when save/import is blocked.
- Keep numerical copy honest: savings and break-even are estimates, never guarantees.
- Persisted settings changes affect **future calculations** after save; communicate this beside the save CTA.
- Restore flow is always three-step and blocking: file chosen → validation/preview shown → explicit full-restore confirmation.
- Invalid backups never mutate existing data and never show a destructive confirmation button.
- Scenario comparison uses fixed order only: konservativ → realistisch → optimistisch → theoretisch.
- Poor data quality must visually downgrade confidence before the scenario values: warning banner above cards, muted secondary explanatory text, no hidden disclaimer.

## Phase-Specific Components

- **Settings card**: two monetary inputs plus one segmented control for `relaxed / balanced / strict`; helper text under each explains future-calculation impact.
- **Tariff period list**: chronological cards with start date, optional end date, Strompreis, Einspeisevergütung, and edit/delete actions.
- **Tariff editor**: date-first form; overlap validation appears before save and names the conflicting period.
- **Backup preview panel**: show schema version, export timestamp, counts for meter readings, PV entries, tariff periods, and whether settings are included.
- **Battery advisor input form**: fields for Speicherpreis, Kapazität, Wirkungsgrad, Betrachtungszeitraum; defaults should visibly prefill rather than hide in placeholders.
- **Scenario result cards**: exactly four cards, same metric order in each card: scenario name → assumed usable share → annual savings → break-even → confidence note.

## State And Feedback Rules

- `loading`: use static skeleton-like cards or muted placeholder values; keep layout stable.
- `empty tariffs`: show empty-state copy plus a visible `Tarifperiode anlegen` secondary action.
- `empty advisor basis`: if not enough analysis data exists, keep scenario cards visible but muted with `Noch keine belastbare Grundlage` instead of blank removal.
- `validation blocked`: first invalid field receives focus after submit, matching existing capture-flow accessibility intent.
- `restore preview ready`: destructive confirmation button stays disabled until preview validation succeeds.
- `restore success`: show calm confirmation banner in accent color, not a celebratory pattern.
- `restore failure` or `invalid backup`: use destructive color and explicitly state that current local data was not changed.
